## ADDED Requirements

### Requirement: local-storage-wfc StorageClass provisioned at cluster-up

The system SHALL provision a StorageClass named `local-storage-wfc` with `WaitForFirstConsumer` volume binding mode and `Delete` reclaim policy when the cluster is brought up. The existing `local-storage` StorageClass (Immediate + Retain) SHALL be retained unchanged.

#### Scenario: StorageClass created at up time

- **WHEN** the user runs `easy-db-lab up`
- **THEN** the system SHALL create a StorageClass named `local-storage-wfc` with `volumeBindingMode: WaitForFirstConsumer` and `reclaimPolicy: Delete`
- **AND** the existing `local-storage` StorageClass SHALL continue to exist with `volumeBindingMode: Immediate` and `reclaimPolicy: Retain`

#### Scenario: StorageClass already exists

- **WHEN** the user runs `easy-db-lab up` and `local-storage-wfc` already exists with the correct configuration
- **THEN** the system SHALL skip creation and proceed normally

### Requirement: Node ordinal labels on db and app node pools

The system SHALL apply `easydblab.com/node-ordinal=N` labels to both db nodes (`type=db`) and app nodes (`type=app`) at cluster-up time. The ordinal is the zero-based index of the node within its pool.

#### Scenario: Ordinal labels applied to db nodes

- **WHEN** the user runs `easy-db-lab up` with db nodes provisioned
- **THEN** the system SHALL label each db node with `easydblab.com/node-ordinal=N` where N is the node's zero-based index within the db pool

#### Scenario: Ordinal labels applied to app nodes

- **WHEN** the user runs `easy-db-lab up` with app (stress) nodes provisioned
- **THEN** the system SHALL label each app node with `easydblab.com/node-ordinal=N` where N is the node's zero-based index within the app pool

#### Scenario: No app nodes provisioned

- **WHEN** the user runs `easy-db-lab up` with zero app nodes
- **THEN** the system SHALL skip app node labeling and proceed normally

### Requirement: platform create-pvs creates per-workload local PVs

The system SHALL provide a `platform create-pvs` command that creates one local PersistentVolume per node in the target node pool, mounted at `/mnt/db1/<workload>`, using the `local-storage-wfc` StorageClass. PVs are created via Fabric8, not via subprocess.

#### Scenario: PVs created for db nodes

- **WHEN** the user runs `easy-db-lab platform create-pvs --workload my-db --size 500Gi`
- **THEN** the system SHALL create one PV per db node at path `/mnt/db1/my-db` with capacity `500Gi` and StorageClass `local-storage-wfc`
- **AND** each PV SHALL have node affinity targeting the node with the matching `easydblab.com/node-ordinal` label

#### Scenario: PVs created for app nodes

- **WHEN** the user runs `easy-db-lab platform create-pvs --workload my-app --size 100Gi --node-type app`
- **THEN** the system SHALL create one PV per app node at path `/mnt/db1/my-app` with capacity `100Gi`

#### Scenario: Existing PVs are not recreated

- **WHEN** the user runs `platform create-pvs` and PVs for the workload already exist and are bound to active PVCs
- **THEN** the system SHALL skip those PVs and emit a warning

#### Scenario: Stale PV claimRef cleared

- **WHEN** a PV exists with a claimRef UID pointing to a deleted PVC
- **THEN** the system SHALL clear the claimRef UID to restore the PV to Available for rebinding

### Requirement: platform info shows substrate details

The system SHALL provide a `platform info` command that displays the K8s substrate configuration: available StorageClasses, PV counts per node pool, node selector keys, anti-affinity key, and example helm values snippet.

#### Scenario: Platform info displays substrate

- **WHEN** the user runs `easy-db-lab platform info`
- **THEN** the system SHALL display StorageClass names and their binding modes, node counts per pool (db, app, control), node selector labels, the ordinal label key, and an example helm values snippet for targeting the substrate

### Requirement: cleanup command performs opt-in disk wipe

The system SHALL provide a `cleanup <workload>` command that creates a K8s Job on each db node to remove `/mnt/db1/<workload>`. This command is never called automatically.

#### Scenario: Cleanup job runs on all db nodes

- **WHEN** the user runs `easy-db-lab cleanup my-db`
- **THEN** the system SHALL create a K8s Job (or DaemonJob equivalent) that runs on each db node and deletes the directory `/mnt/db1/my-db`

#### Scenario: Cleanup is never automatic

- **WHEN** the user runs a `stop.sh` script generated by `install`
- **THEN** disk data at `/mnt/db1/<workload>` SHALL be preserved
- **AND** the system SHALL NOT invoke `cleanup` automatically
