# Workload Metrics Catalog

## Overview

A standard set of artifacts and tooling for discovering, documenting, and visualizing metrics from scrape-type workloads. Anchored to real data from a live cluster: a global export script queries VictoriaMetrics, producing a committed JSON catalog that drives both a machine-readable agent spec (`METRICS.md`) and a Grafana dashboard.

## ADDED Requirements

### Requirement: Global export script queries VictoriaMetrics for any scrape-type workload
A single script `bin/export-workload-metrics` SHALL accept a workload name as its only argument, read `CONTROL_HOST_PRIVATE` from `env.sh` in the current directory, query `http://$CONTROL_HOST_PRIVATE:8428/api/v1/series?match[]={job="<workload>"}`, normalize the response, and write `<workload>/metrics-catalog.json` in the current cluster working directory.

#### Scenario: Export produces metrics-catalog.json for a running workload
- **WHEN** a scrape-type workload is running and `bin/export-workload-metrics presto` is executed from the cluster working directory
- **THEN** a file `presto/metrics-catalog.json` SHALL be written with keys `workload`, `exported_at` (ISO-8601 UTC), and `series` (array of objects with `name` and `labels` fields)
- **AND** the `series` array SHALL contain at least one entry

#### Scenario: Export script fails if workload argument is missing
- **WHEN** `bin/export-workload-metrics` is called with no arguments
- **THEN** the script SHALL print usage and exit non-zero

#### Scenario: Export script fails if env.sh is not present
- **WHEN** `bin/export-workload-metrics presto` is called from a directory with no `env.sh`
- **THEN** the script SHALL print an error and exit non-zero

### Requirement: Committed metrics-catalog.json inside workload resource templates
Each scrape-type workload's resource directory (`src/main/resources/.../install/<workload>/`) SHALL contain a committed `metrics-catalog.json` that was generated by `bin/export-workload-metrics` against a real cluster. This file is the authoritative source for METRICS.md and the Grafana dashboard.

#### Scenario: metrics-catalog.json is present for presto workload
- **WHEN** the presto workload resource directory is inspected
- **THEN** a `metrics-catalog.json` file SHALL exist containing at least 10 series entries

#### Scenario: metrics-catalog.json is present for clickhouse workload
- **WHEN** the clickhouse workload resource directory is inspected
- **THEN** a `metrics-catalog.json` file SHALL exist containing at least 10 series entries

### Requirement: METRICS.md is a machine-readable agent spec per workload
Each scrape-type workload's resource directory SHALL contain a `METRICS.md` file with a markdown table of key metrics (name, relevant labels, description). The file is authored by a human from the catalog and is intended to be read by agents, not displayed as user documentation.

#### Scenario: METRICS.md lists key metrics for presto
- **WHEN** `METRICS.md` for the presto workload is read
- **THEN** it SHALL contain a markdown table with columns `Metric`, `Labels`, and `Description`
- **AND** all metric names in the table SHALL appear in `metrics-catalog.json`

#### Scenario: METRICS.md lists key metrics for clickhouse
- **WHEN** `METRICS.md` for the clickhouse workload is read
- **THEN** it SHALL contain a markdown table with columns `Metric`, `Labels`, and `Description`
- **AND** all metric names in the table SHALL appear in `metrics-catalog.json`

### Requirement: Grafana dashboard committed per workload, built from real catalog data
Each scrape-type workload's resource directory SHALL contain `dashboards/<workload>.json` — a Grafana dashboard JSON built from metric names in `metrics-catalog.json`. Dashboards SHALL include a `cluster` multi-select variable and SHALL scope all panel queries with `{cluster=~"$cluster",job="<workload>"}`. The dashboard is authored once and committed; it is not generated at runtime.

#### Scenario: Presto dashboard is installed when presto starts
- **WHEN** `easy-db-lab install presto start` completes successfully
- **THEN** a Grafana dashboard named "Presto" SHALL appear in Grafana
- **AND** all panels SHALL query the VictoriaMetrics datasource

#### Scenario: ClickHouse workload dashboard is installed when clickhouse starts
- **WHEN** `easy-db-lab install clickhouse start` completes successfully
- **THEN** a Grafana dashboard named "ClickHouse (Workload)" SHALL appear in Grafana (distinct from any built-in ClickHouse dashboard)
- **AND** all panels SHALL query the VictoriaMetrics datasource

### Requirement: Workload-creation skill documents the metrics catalog workflow
The `generate-workload` skill (or equivalent workload-creation guidance) SHALL document that scrape-type workloads SHOULD include `metrics-catalog.json`, `METRICS.md`, and a dashboard, and SHALL explain the workflow for generating them from a live cluster.

#### Scenario: Skill explains the catalog-first workflow
- **WHEN** the workload-creation skill is invoked for a scrape-type workload
- **THEN** the generated guidance SHALL mention `bin/export-workload-metrics`
- **AND** SHALL explain that METRICS.md and the dashboard are authored from the catalog, not generated automatically
